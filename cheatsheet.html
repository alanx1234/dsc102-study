Algorithm Comparison
  • FCFS: simple, no starvation; worst avg TAT if long job first (convoy)
  • SJF: optimal avg TAT (non-preemptive); starvation risk for long jobs
  • SRTF: optimal avg TAT (preemptive); best for TAT, high overhead
  • RR: best avg response time; fair; bad TAT if q too large or small

Response time ranking (best→worst): RR ≥ SRTF > SJF > FCFS
Turnaround ranking (best→worst): SRTF = SJF > RR > FCFS

Scheduling Algorithms

FCFS: treat the ready queue as a literal FIFO — process at the front runs until done, then dequeue next. Convoy effect: one long job stalls all shorter ones behind it.

SJF (non-preemptive): when CPU becomes free, scan all READY jobs and pick the one with the shortest burst. Does NOT interrupt a currently running job even if something shorter arrives.

SRTF (preemptive SJF): on every new arrival (and on each completion), compare the newcomer's burst to the running job's REMAINING time. If newcomer is shorter → preempt immediately. Think: always running the minimum remaining time in the ready set.

Round Robin: cycle through jobs giving each a fixed quantum q. Unfinished job goes to back of queue. Good response time. If q too small → context-switch overhead dominates; if q too large → degenerates to FCFS.

Disk I/O — HDD Mechanics

Three components of every I/O:

  • Seek: arm moves to correct track (1–20ms, avg ~4–8ms)
  • Rotational latency: wait for sector to spin under head (avg = half rotation)
  • Transfer: data moves off platter (~100s MB/s, small per-op)
rot_period = 60,000ms / RPM avg_rot = rot_period / 2 T_IO = seek + avg_rot + transfer transfer = data_size / bandwidth bandwidth = IOPS × IO_size

Sequential vs random: sequential can be 100–300× faster than random because contiguous blocks skip seeks and rotational waits.

Disk I/O — Contiguous vs Chunked
Contiguous file: T = 1×seek + 1×avg_rot + file_size/BW Chunked access (n chunks of size s): chunk_transfer = s / BW T_per_chunk = seek + avg_rot + chunk_transfer T_total = n × T_per_chunk

Slide example — 10GB, 7200RPM, seek=8ms, BW=150MB/s, 512KB chunks:
rot=60000/7200=8.33ms → avg_rot=4.17ms
chunk_transfer=0.5/150=0.00333s=3.33ms
T_per_chunk=8+4.17+3.33=15.5ms
n_chunks=10240MB/0.5MB=20,480
Total=20480×15.5ms=317s≈5.3min
Sequential only: 10240/150=68s → chunked is 4.7× slower

Contiguous vs fragmented — 2GB, 7200RPM, seek=8ms, BW=150MB/s, 512 frags:
avg_rot=4.17ms | transfer=2048/150=13.65s
Contiguous: 12.17ms+13.65s≈13.66s
Fragmented: 512×12.17ms+13.65s≈19.88s (1.46× slower)

Storage Performance Terms
  • IOPS: I/O operations per second the device can sustain
  • Bandwidth: bytes/sec = IOPS × IO_size
  • Latency: total time to deliver data (seek+rot+transfer+OS delays)

40GB file, disk BW=800MB/s: 40×10⁹/8×10⁸=50s
Same over 1.6Gbps network (=200MB/s): 40×10⁹/2×10⁸=200s

SSD vs HDD: SSD has no seek/rotation → 100× faster random I/O, 10–100× higher BW. Still block-addressable.

Disk Scheduling

SSTF (Shortest Seek Time First): pick pending request closest to current head position. Fast but can starve far-away tracks.

SCAN/C-SCAN (elevator): sweep arm in one direction serving requests, then reverse (or jump back). Prevents starvation; ignores rotational latency.

SPTF (Shortest Positioning Time First): account for BOTH seek AND rotation — pick the request with least total positioning time. Best in practice; done inside drive firmware today.

Multi-core Scheduling

SQMS: one shared queue, any core picks next. Simple; lock contention at scale.

MQMS: one queue per core. Less contention; risk of load imbalance.

Makespan = time until last job finishes = max(sum of jobs per core)
Row-Store vs Column-Store

Row-store: all columns of a row stored together. Good for full-row reads (OLTP). Bad for column-only queries — must scan all columns.

Column-store (Parquet, ORC): each column stored together. Great for column-subset queries — only read needed columns. Better compression (same-type data together).

Parquet vs CSV: compressed (up to 10× smaller), column pruning, schema+metadata embedded, immutable. Cannot edit in-place.

1TB CSV → 130GB Parquet; query 236s→6.78s (34×); scans 99% less data.

NEW → READY → RUNNING → BLOCKED → READY

  • RUNNING→BLOCKED: process issues I/O syscall
  • BLOCKED→READY: I/O completes
  • RUNNING→READY: timer interrupt (preemption)
  • Threads share address space; processes don't
  • GIL: CPython = 1 thread at a time (CPU-bound bottleneck)
OS Mechanics — How It Actually Works

User mode vs Kernel mode: Hardware enforces two privilege levels. Normal programs run in user mode (can't touch hardware directly). OS runs in kernel mode (full access).

Syscall (system call): only way for user program to request OS services. Program executes a trap instruction → CPU switches to kernel mode → OS handles the request → returns to user mode. Examples: read(), write(), fork(), malloc() (which calls brk/mmap).

Timer interrupt: hardware timer fires ~every 10ms. CPU automatically jumps to OS interrupt handler (in kernel mode). OS can then: (1) preempt current process (RUNNING→READY), (2) pick another process, (3) return. This is how the OS regains control without process cooperation.

Context switch — step by step:

  1. Timer fires → CPU saves PC + registers onto kernel stack
  2. OS saves them into the outgoing process's PCB
  3. Scheduler picks next process from ready queue
  4. OS loads new process's PCB (registers, PC) → switch page table (new VA space)
  5. Return from interrupt → new process resumes in user mode

PCB (Process Control Block): per-process OS struct storing registers, PC, state, open files, memory map, PID. Lives in kernel memory.

VM Translation Flow (Conceptual)

Why VM exists: Each process gets illusion of its own private address space (isolation). OS+hardware prevent one process from touching another's memory. Allows using disk as overflow when RAM is full.

Every memory access — what happens:

  1. CPU generates virtual address (VA = VPN + offset)
  2. MMU checks TLB (hardware cache of recent VPN→PFN translations, ~1 cycle)
  3. TLB HIT: get PFN → PA = PFN × page_size + offset. Done.
  4. TLB MISS: MMU walks page table in memory → load PFN into TLB → retry
  5. Page FAULT: PTE says "not in RAM" → trap to OS → OS finds page on disk → loads into free frame → updates PTE → TLB loaded → resume

TLB: tiny fast associative cache on the CPU die (~32–1024 entries). Stores recently used VPN→PFN pairs. Flushed on context switch (different process = different VA space).

Page fault cost: disk access = millions of cycles. So thrashing (constant faults) kills performance.

Valid bit in PTE: 0 = page not in RAM (trigger fault). 1 = page present, PFN valid. Dirty bit: 1 = page was written → must write back to disk before evicting.

NUMA & False Sharing

NUMA (Non-Uniform Memory Access):

  • Multi-socket machines: each socket has "local" RAM
  • Accessing remote socket's RAM = k× slower than local

Local=100ns, remote=3× slower:
Thread on Socket 0 reading Socket 1's memory → 300ns

False Sharing:

  • Two threads write different variables on the same cache line
  • Every write forces other core to invalidate its copy → constant line bouncing
  • Fix: pad struct so each thread's variable is on its own cache line

Thread A writes x, Thread B writes y. x and y are adjacent ints (4B each) on same 64B cache line → false sharing → repeated cache invalidations even though they write to different variables.

True sharing: threads actually share the same variable (needed for coordination).

Parallelism Concepts

SIMD: single instruction, multiple data — one instruction on a vector of values (e.g., add 8 floats at once). Used in NumPy, GPU.

GPU vs CPU: GPU = thousands of simple cores for data-parallel work; CPU = few powerful cores for complex branchy code.

Why NumPy beats Python loops: contiguous typed memory, compiled C kernels, SIMD vectorization, no per-element Python overhead.

Parallel overhead types: (1) Communication: cores share results (network/memory bus). (2) Synchronization: waiting at barriers/locks. (3) Load imbalance: fast workers idle waiting for slow ones. All reduce efficiency below 1.0.

Locks & Synchronization

Race condition: two threads read/modify same shared data concurrently. Outcome depends on timing. Classic: both read x=0, both do x++, both write x=1 → lost update (should be 2).

Critical section: code region that must run atomically — only one thread at a time. Solution: protect with a lock.

Mutex (lock): binary flag. thread calls lock() to enter — if already held, the caller sleeps (blocks). Holder calls unlock() when done → one waiter wakes up.

Spinlock: thread busy-loops checking the lock (burns CPU). Use when wait is very short (saves context-switch overhead). Bad if wait is long.

Deadlock: Thread A holds Lock 1, needs Lock 2. Thread B holds Lock 2, needs Lock 1. Both block forever.

Fix deadlock: always acquire locks in the same global order (e.g., always lock 1 before lock 2).

Atomic operations: hardware guarantee — read-modify-write completes without interruption (compare-and-swap, fetch-and-add). No lock needed for simple counters.

Condition variable: lets a thread sleep until a condition is true (e.g., buffer non-empty). Used with a mutex: unlock mutex → sleep → re-lock on wakeup.

Amdahl: Solving for p or n
Given speedup S and n workers: solve for p S = 1/((1−p) + p/n) 1/S = (1−p) + p/n = 1 − p + p/n = 1 − p(1 − 1/n) p = (1 − 1/S) / (1 − 1/n) Given speedup S and p: solve for n 1/S = 1 − p + p/n p/n = 1/S − 1 + p n = p / (1/S − 1 + p)

Observed speedup=3× with 6 cores: p=(1−1/3)/(1−1/6)=(2/3)/(5/6)=0.8 (80% parallel)

MCQ Concept Keys
  • Scheduling goal: maximize CPU util / multiprogramming — NOT speed up one process
  • Response time low = important for interactive jobs. Batch jobs → minimize turnaround
  • Processes don't share virtual addr space. DO share: physical RAM, CPU, disk
  • Cache hit rate primarily set by cache SIZE (larger → fewer misses). Not DRAM speed
  • Timer interrupt = how OS reclaims CPU from running process without its cooperation
  • "2's complement of 12" = representation of −12: flip(00001100)+1 = 11110100
  • ABCD∞ LRU 3 frames: 100% fault rate — 4 unique pages > 3 frames, always evicts
  • ABCD∞ OPT 3 frames: ≈33% fault rate — OPT can load ABC, strategically defer D
  • fork() child: child=0 in child process, child=PID in parent. Both resume after fork()
  • Larger pages: ↑ internal frag, ZERO external frag (paging always uses fixed-size frames)
  • Thrashing → ↓ CPU utilization (CPU spending all time on page faults, not useful work)
  • SJF: selects shortest at time CPU is free. Does NOT preempt running job even if shorter arrives
  • SRTF preempts when new arrival's burst < current job's REMAINING time (not original burst)
